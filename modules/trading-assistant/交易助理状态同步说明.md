# 交易助理状态同步说明

## 目标

本地与 GitHub Pages 必须共享同一份业务连续性状态，避免本地旧行情快照覆盖云端已经刷新过的候选、推荐和复盘记录。

GitHub Pages 仍然是静态页面。策略确认、候选剔除等写入动作只能在本地服务 `http://127.0.0.1:8765/` 执行；本地服务会在执行前拉取云端状态，执行后自动发布状态。

## 同步范围

同步分支为 `trading-assistant-state`。以下文件是唯一需要跨本地、GitHub Actions 和 Pages 连续保存的状态：

| 文件 | 保留的信息 |
| --- | --- |
| `trading-assistant.json` | 当前候选股票、行动清单、市场与板块分析、复盘摘要、页面展示快照。 |
| `trading-assistant-recommendation-tracking.json` | 所有曾推荐股票、推荐日期、价格轨迹、持仓建议、是否仍在跟踪、收益回看。 |
| `trading-assistant-strategy-log.json` | 每次刷新形成的选股与交易判断日志，用于日/周/月复盘。 |
| `trading-assistant-strategy-upgrade-state.json` | 已确认的策略升级、执行效果与回退历史。 |
| `trading-assistant-removal-state.json` | 候选池剔除确认、继续跟踪确认及其理由。 |

以下文件不属于业务连续性状态，不同步到状态分支：

- `trading-assistant-universe-cache.json`：临时股票池缓存，刷新时可重新获得。
- `trading-assistant.js`：由 `trading-assistant.json` 自动生成的浏览器快照。
- 临时构建目录 `dist/`。

## 运行流程

1. GitHub Actions 每个交易日三次启动，先从 `trading-assistant-state` 恢复业务状态。
2. Actions 获取行情、K线、财报、公告等真实数据，重新计算候选股票、交易计划、推荐追踪和复盘。
3. Actions 把新的业务连续性状态写回 `trading-assistant-state`，然后部署 GitHub Pages。
4. 用户在本地确认策略升级、回退、剔除或保留跟踪时，本地服务先恢复云端状态，再执行操作，最后发布完整状态到该分支。
5. 本地发布状态后会创建一个只含时间戳的 `main` 分支部署信号，由 `main` 发起 GitHub Pages 部署，满足 Pages 环境保护规则。由 Actions 自己写回的状态提交不会触发循环执行。

## 本地命令

```powershell
npm run state:pull:trade
npm run state:push:trade
```

通常无需手动运行。它们用于网络恢复后补发状态，或排查云端同步失败。

## 失败处理

- 本地确认操作已经成功，但云端同步失败：页面会明确显示“本地已更新，云端同步失败”，不会伪装成线上已生效。网络恢复后重新执行对应操作即可，操作按策略键和股票代码去重。
- GitHub Actions 数据源失败：保留最近一次成功的业务状态，并在页面刷新状态中列出失败数据源和替代来源。
- 状态分支尚未创建：首次本地发布会创建该分支；首次云端刷新会使用仓库内初始数据并随后写入状态分支。
