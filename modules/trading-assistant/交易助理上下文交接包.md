版本：v0.2  
最后整理日期：2026-07-20  
适用场景：把本文档交给其它 AI、工程师或后续开发线程，用来复刻或继续开发一个类似的 A 股交易助理系统。  
当前模块目录：`modules/trading-assistant`  
当前线上地址：https://frankkeqiao-png.github.io/stock-ai-assistant/trading-assistant.html  
GitHub 仓库：https://github.com/frankkeqiao-png/stock-ai-assistant.git

> 本系统只做股票决策辅助，不替用户下单，不构成投资建议。所有推荐必须展示数据、逻辑、风险、下一步验证条件，禁止编造行情、财报、公告、资金流等数据。

## 1. 项目目标

用户最终想做的是一个大型股票 AI 助理系统，长期会包含市场阶段、宏观政策、全球映射、产业链、公司情况、财报公告、技术面、候选池、复盘、自我修复等模块。

当前为了保证效果，先单独开发其中一个子模块：交易助理。

交易助理的核心目标是：

1. 帮用户从 A 股中筛选值得关注的股票。
2. 给出每只股票为什么进入候选池、为什么值得看、为什么暂不交易。
3. 给出接近实战的操作指导：买入时机、建仓区间、首仓比例、加仓条件、止损、止盈、下一触发条件。
4. 使用技术面、缠论买卖点、趋势强弱、风险收益比、资金流、财报、公告、板块归因等多维度综合判断。
5. 每天自动刷新真实数据，不只是填充页面，而是重新计算候选池、行动清单、买卖点、交易计划和推荐追踪。
6. 对历史推荐进行连续追踪，方便用户检查推荐是否经得起时间验证。
7. 推荐池不能每天大幅换股。股票操作不是一天两天的超短线，因此要区分中期跟踪和日内新机会。
8. 任何股票移出推荐追踪池前，必须给出理由并等待用户确认。

## 2. 模块边界

交易助理是独立子模块，后续开发其它模块时不能覆盖或混入本模块。

## 4. 用户偏好和板块范围

用户当前偏好：

1. 科技板块权重最高。
2. 其它板块也关注，但每个板块候选数量少于科技。
3. 具体选股和公司分析聚焦保留板块；宏观或全市场模块未来可以全面覆盖。
4. 不只因为命中板块就推荐，必须满足趋势、买点、风险收益比、基本面、公告等条件。

当前保留板块：

| 板块 | 权重/优先级 | 当前候选数量目标 |
|---|---:|---:|
| 科技 | 最高 | 约 16 只 |
| 新能源/电力设备 | 高 | 约 3 只 |
| 机器人/智能制造 | 中高 | 约 3 只 |
| 低空经济/航空航天 | 中高 | 约 3-4 只 |
| 高端制造/汽车 | 中高 | 约 3 只 |
| 医药医疗 | 中 | 约 3 只 |
| 军工国防 | 中 | 约 3-4 只 |
| 黄金/贵金属 | 中 | 约 3 只 |
| 资源周期/化工 | 中 | 约 3 只 |
| 大消费 | 低于科技 | 约 3 只 |

明确排除板块：

```text
金融、房地产、建筑、外贸、农业/养殖、公用环保/交通、教育/服务
```

这些板块不进入交易助理的候选池和推荐池。

## 5. 页面结构

当前 Web 页面包含 8 个页面：

| 页面 | 名称 | 解决的问题 |
|---|---|---|
| 01 | 今日行动清单 | 今天重点看什么，哪些继续跟踪，哪些是新机会。 |
| 02 | 候选股票 | 哪些股票进入候选池，按板块展示，解释入池理由、评分、风险和下一步。 |
| 03 | 买卖点触发 | 用缠论/技术面判断是否接近买点，展示建仓区间、止损、止盈、触发条件。 |
| 04 | 交易计划 | 每只候选股的具体操作计划：首仓、加仓、止损、止盈、下一动作。 |
| 05 | 数据复盘 | 数据覆盖、数据源状态、日/周/月复盘、策略升级建议。 |
| 06 | 剔除确认 | 对拟移出推荐池的股票给出理由，用户确认后才停止追踪。 |
| 07 | 推荐追踪 | 历史推荐从推荐日起到当前的走势、收益、回撤、是否触发止损止盈。 |
| 08 | 全量数据 | 展示结构化 JSON，方便二次判断和排错。 |

界面要求：

1. 每个页面顶部显示刷新时间，格式为 `YYYY年MM月DD日 HH:mm`。
2. 自动刷新或手动刷新后的结果必须有状态反馈。
3. 如果有失败源，显示失败源数量；展开后列出具体失败源、替代方案和影响。
4. 大块审计信息、数据源状态、候选池来源等用折叠方式展示，避免页面过长。
5. 候选股票页面按板块分组，科技放最前面。
6. 每只股票必须显示理由、逻辑、数据、下一步建议，不能只给结论。
7. GitHub Pages 静态模式下，页面按钮只能重新读取已发布数据；真正刷新由 GitHub Actions 执行。

## 6. 数据源和降级原则

系统必须使用真实公开数据，不允许伪造。

当前数据源设计：

| 数据类型 | 主源 | 备用/补位 |
|---|---|---|
| 板块和股票池 | 东方财富板块、行业、概念 | 新浪财经行业/概念节点、本地股票池缓存 |
| 个股实时行情 | 东方财富/新浪行情 | 新浪财经、腾讯行情补位 |
| 日 K、30 分钟 K、5 分钟 K | 东方财富 K 线 | 新浪财经 K 线、腾讯证券 K 线 |
| 财报 | 东方财富财务报表 | 暂无稳定全量替代，缺失时显性标注 |
| 公告 | 东方财富公告 | 巨潮资讯公告备用源 |
| F10/公司信息 | 东方财富 F10 | 可扩展其它公开公司资料源 |
| 资金流 | 东方财富资金流 | 新浪资金流备用源；仍缺失时不加分不扣分 |
| 换手率补位 | 东方财富行情字段 | 腾讯证券补位 |

降级原则：

1. 东方财富失败时，优先尝试新浪财经或腾讯备用源。
2. 如果板块官方涨跌幅/成交额不可用，可以用成分股真实行情聚合计算。
3. 如果资金流缺失，必须显示“资金流缺失”，该项不加分也不扣分。
4. 如果财报、公告、K 线缺失，不能假设正常，必须显示缺失并降低判断强度。
5. 每次刷新后，失败源必须进入 `audit.failures`，并在页面黄色状态框和数据复盘页展示。

## 7. 选股总逻辑

股票不能只因为命中用户关注板块就推荐。推荐逻辑必须是多条件合成。

基本判断链：

```text
板块方向匹配
+ 股票池流动性满足
+ 趋势结构可观察
+ 缠论买点接近或确认
+ 量价关系健康
+ 相对强度较好
+ 财报基本面没有明显硬伤
+ 公告和事件没有重大负面
+ 资金行为不冲突，或缺失时显性标注
+ 风险收益比合适
+ 能形成明确交易计划
= 进入候选/重点跟踪/交易准备
```

主要评分维度：

| 维度 | 作用 |
|---|---|
| 板块归因 | 判断股票是否属于用户关注方向，科技权重最高。 |
| 流动性 | 成交额过低会降低执行可行性。 |
| 当日表现 | 当日强弱可加减分，但不能单独决定推荐。 |
| 换手和量比 | 判断关注度和短线分歧。 |
| 估值 | 极端高估值扣分，但不做唯一否决。 |
| 资金流 | 主力净流入是重要参考，但缺失不改变判断逻辑。 |
| 财报质量 | 营收、净利润、毛利、现金流、负债等。 |
| 公告事件 | 正面催化加分，负面公告/风险提示扣分。 |
| 技术趋势 | 日线趋势阶段、均线、支撑压力、波动位置。 |
| 缠论结构 | 类二买、类三买、潜在一买、无买点/风险段等。 |
| 多级别确认 | 日线看方向，30 分钟和 5 分钟看触发。 |
| 风险收益比 | 买入区、止损、止盈计算出来的收益/风险比。 |

## 8. 交易状态分层

交易助理有两套分层：交易状态和推荐稳定层。

### 8.1 交易状态

| 状态 | 含义 | 操作建议 |
|---|---|---|
| 交易准备池 | 评分、趋势、买点、风险收益比基本满足 | 等触发条件确认后可考虑首仓。 |
| 重点跟踪池 | 方向较好，但买点或低级别确认仍不足 | 继续观察，等待回踩、突破或低级别确认。 |
| 观察池 | 有一定逻辑，但条件不足 | 保留观察，不执行交易。 |
| 暂不交易 | 技术、基本面、公告或风险收益比存在硬约束 | 不交易，等待修复或后续剔除。 |
| 低优先级 | 分数过低或条件弱 | 仅保留在底层观察。 |

### 8.2 推荐稳定层

这是后来为解决“每天推荐差异过大”而新增的机制。

| 稳定层 | 含义 |
|---|---|
| 核心跟踪池 | 已经经过上一轮验证，继续值得跟踪，不因单日排序波动移出。 |
| 今日机会池 | 今天新发现、条件较强的机会，同日内不升级为核心跟踪。 |
| 候选观察池 | 有逻辑但暂未形成稳定交易机会。 |
| 待确认剔除 | 原核心/重点候选恶化，暂停新开仓，等待用户确认是否移出追踪。 |
| 暂不交易 | 当前交易条件不足。 |

关键规则：

1. 原来属于交易准备池、重点跟踪池、高分观察池、核心跟踪池或用户手动保留的股票，本次刷新会优先带入重新分析。
2. 同一交易日内新发现的机会保持“今日机会池”，至少跨交易日继续满足条件后，才升级为“核心跟踪池”。
3. 普通“暂不交易”弱候选不会占用剔除确认列表。
4. 只有真正推荐过或值得跟踪的股票恶化后，才进入“待确认剔除”。
5. 未经用户确认，历史推荐不会停止追踪。

## 9. 技术面和缠论框架

系统要吸收用户提供的缠论资料，并将缠论作为技术面判断的一部分。

当前技术面要素：

1. 日线趋势阶段：趋势上升、震荡、破位、风险段等。
2. 均线结构：MA5、MA10、MA20、MA60 等。
3. 价格位置：高低点、20 日高低、支撑压力。
4. 动量：20 日、60 日、120 日涨跌幅。
5. 候选池相对强度：用候选池内部排名近似 RPS。
6. 30 分钟 K：用于中级别确认。
7. 5 分钟 K：用于更低级别买点触发。
8. 缠论买点：
   - 潜在一买观察
   - 类二买/趋势延续观察
   - 类三买观察
   - 中枢上半区观察
   - 无买点/风险段
9. 输出必须包含买点类型、是否确认、建仓区间、止损、止盈、触发条件、失效条件。

技术面输出不是“可以买/不能买”这么简单，必须描述：

```text
当前结构是什么
买点是否已经确认
如果未确认，下一触发条件是什么
如果确认，合理建仓区间是什么
跌破哪里计划失效
上方目标或止盈观察位在哪里
风险收益比是否达标
```

## 11. 推荐追踪机制

推荐追踪页的目标是让用户看到“系统过去推荐的股票是否经得起时间验证”。

规则：

1. 每次刷新后，当前候选池的股票都写入推荐追踪。
2. 推荐追踪记录股票从首次推荐日起的价格走势。
3. 展示每只股票的推荐日期、推荐价格、当前价格、累计收益、最大涨幅、最大回撤、是否触发止损或止盈观察。
4. 推荐追踪里的股票不因为当天未进入候选池就删除。
5. 如果系统建议踢出，必须进入“剔除确认”页。
6. 用户确认踢出后，该股票在推荐追踪页变灰并放到最后，不删除历史记录。

当前审计口径：

```text
当前候选必须全部进入推荐追踪。
历史推荐在未经人工确认剔除前可以继续 active。
因此 activeTrackingCount 允许大于 candidateCount。
```

## 12. 剔除确认机制

剔除不是自动执行，而是建议 + 用户确认。

进入待确认剔除的条件：

1. 原来属于核心跟踪、交易准备、重点跟踪、高分观察或用户手动保留。
2. 本次刷新出现硬风险：
   - 暂不交易或低优先级；
   - 趋势进入明显风险段；
   - 风险收益比低于最低要求；
   - 财报、公告或事件出现重大负面；
   - 跌破止损或结构失效。

不进入待确认剔除的情况：

1. 昨天只是普通暂不交易候选，今天仍然弱。
2. 只是单日排序下降，但趋势和计划仍然有效。
3. 数据源缺失但未证明交易逻辑失效。

用户操作：

| 操作 | 结果 |
|---|---|
| 确认剔除 | 写入 `confirmedRemoved`，推荐追踪停止 active，页面变灰放最后。 |
| 继续跟踪 | 写入 `keepTracking`，后续刷新强制带入重新分析。 |

## 13. 复盘和策略升级

用户要求系统具备“自我修复机制”，但不能过度精确化。

设计约束：

1. 复盘不是每天只要没达到预期就改策略。
2. 要按一段时间看，如日、周、月、季度、半年。
3. 股票偏差可能来自不可知因素，不能简单归因于策略错误。
4. 目标是保证大方向和结论逐步更合理，而不是追求单笔预测绝对准确。
5. 复盘发现需要升级策略时，必须列出建议并让用户点击确认。
6. 用户确认后才能记录策略升级，不允许自动改核心策略。

数据复盘页应包含：

1. 分层表现基线。
2. 本次观察。
3. 日复盘、周复盘、月复盘，按日期折叠展示。
4. 数据覆盖和失败源。
5. 策略升级建议，每条显示提出日期。
6. “确认升级”按钮。

触发条件：

1. push 到 `main` 或 `master`。
2. 手动运行 workflow。
3. 工作日定时运行三次。

GitHub Actions cron 使用 UTC：

```yaml
schedule:
  - cron: "10 1 * * 1-5"
  - cron: "45 4 * * 1-5"
  - cron: "40 7 * * 1-5"
```

对应北京时间：

```text
09:10
12:45
15:40
```

GitHub Pages 模式限制：

1. 页面是静态页面，不能直接写入本地确认状态。
2. 页面刷新按钮只能重新读取已经发布的 `data/trading-assistant.json`。
3. 真正的数据刷新由 GitHub Actions 执行。
4. 如果需要确认剔除、确认策略升级，建议在本地运行后处理，再提交推送。

## 15. GitHub 开源项目借鉴情况

曾评估过若干高使用度或常见量化/交易相关开源项目，但当前系统没有直接复制或 vendoring 这些项目代码。主要是吸收设计思想。

| 项目/类型 | 可借鉴点 | 当前是否直接使用 | 未直接使用原因 |
|---|---|---|---|
| Microsoft Qlib | 研究流水线、数据集、模型评估、策略复盘思路 | 否 | 系统当前更偏 Web 交易助理和可解释规则，不做复杂 ML 训练。 |
| Backtrader | 回测框架、订单和策略生命周期概念 | 否 | 当前不下单、不做完整回测引擎；只保留“计划、追踪、复盘”的思想。 |
| vectorbt | 向量化回测、指标评估、批量策略验证 | 否 | 当前数据源和用户需求更偏实时候选解释，未引入 Python 回测栈。 |
| vn.py | 实盘交易系统架构、行情/交易分离 | 否 | 用户明确不需要代下单，交易接口和券商接入暂不做。 |
| TA-Lib / pandas-ta | 技术指标计算思路 | 否 | 当前用原生 JS 实现轻量均线、动量、买点判断，减少依赖。 |
| AkShare / TuShare 思路 | A 股公开数据源和数据统一层 | 未正式接入 | TuShare token 对用户复杂；AkShare 需要 Python 依赖，当前优先 JS + 公开接口。 |

吸收后的设计取舍：

1. 保留“策略可复盘”的思想。
2. 保留“候选池和推荐追踪分离”的思想。
3. 保留“数据源失败要显性审计”的思想。
4. 不引入黑盒机器学习推荐，优先可解释规则。
5. 不做自动下单、券商交易、仓位实盘同步。

## 17. 重要策略约束，后续 AI 不要随意破坏

1. 不允许伪造数据。
2. 资金流是重要参考，但公开源失败时不降级逻辑、不编造值。
3. 推荐不是每天全量换股；必须保持连续性。
4. 今日机会不能同日多次刷新后立刻升级为核心跟踪。
5. 普通暂不交易弱候选不进入剔除确认，避免用户确认列表过长。
6. 真正推荐过或值得跟踪的股票，恶化后才进入待确认剔除。
7. 推荐追踪里的历史股票不能直接删除。
8. 用户确认前不能执行策略升级。
9. 科技板块权重最高，但其它保留板块也要覆盖。
10. 禁用板块不能出现在候选池。
11. 每只股票输出必须包含理由、数据、逻辑、风险和下一步建议。
12. GitHub Pages 静态模式不能写入确认状态。

## 18. 当前已知问题和后续建议

1. 当前系统主要是规则型交易助理，还没有完整历史回测引擎。
2. 资金流公开接口稳定性不足，已做缺失显性提示，但未来可寻找更稳定免费源。
3. 财报分析目前基于公开报表摘要，未来可增加同比/环比深度、行业对比和财务质量评分。
4. 缠论判断目前是工程化近似，不等于人工缠论专家的完整画笔、线段、中枢识别。
5. 推荐准确率需要靠推荐追踪和复盘逐步验证。
6. 未来大系统集成时，可以把宏观、产业链、公司深度分析作为上游输入，影响交易助理的加分/扣分和风险说明。
