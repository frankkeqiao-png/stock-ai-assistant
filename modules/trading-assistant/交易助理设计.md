# 交易助理设计

版本：v0.1  
适用对象：A股交易决策辅助系统中的“交易助理”子模块  
定位：不替用户下单，只提供选股、候选池、买卖点、交易计划、持续追踪、复盘和数据审计辅助。

## 1. 模块目标

交易助理模块的核心目标是帮助用户完成以下工作：

1. 从A股市场中筛选值得关注的股票。
2. 给出每只股票进入候选池、重点跟踪池、交易准备池或暂不交易的理由。
3. 对候选股给出接近实战的操作指导，包括买入时机、建仓区间、首仓比例、加仓条件、止损、止盈、下一步观察条件。
4. 使用技术面、缠论结构、趋势强弱、资金行为、财报、公告、行业归因等多维度形成综合判断。
5. 每次刷新数据时动态调整候选池、交易计划、行动清单和推荐追踪记录。
6. 对历史推荐进行持续跟踪，便于用户检查推荐是否经受时间检验。
7. 在数据源失败时显性提示失败源，不伪造数据。
8. 保持模块独立，后续可与宏观、产业链、公司分析等其它模块集成。

## 2. 用户偏好与筛选范围

用户当前偏好：

- 科技板块权重最高。
- 其它保留板块也参与筛选，但每个板块数量少于科技板块。
- 选股时不只看板块命中，还必须满足趋势、技术、基本面、风险收益比等条件。

当前保留板块：

| 板块 | 目标数量 | 说明 |
|---|---:|---|
| 科技 | 约16只 | 半导体、芯片、AI、算力、软件、通信、电子、PCB、存储、数据中心等 |
| 新能源/电力设备 | 约3只 | 锂电、储能、光伏、固态电池、电网、电力设备等 |
| 机器人/智能制造 | 约3只 | 机器人、人形机器人、自动化、数控、工业母机等 |
| 低空经济/航空航天 | 约3只 | 低空经济、无人机、航空、航天、卫星等 |
| 高端制造/汽车 | 约3只 | 新能源车、汽车零部件、机械、设备、仪器仪表等 |
| 医药医疗 | 约3只 | 创新药、医疗器械、中药、生物医药等 |
| 军工国防 | 约3只 | 军工、国防、北斗、卫星、航空航天等 |
| 黄金/贵金属 | 约3只 | 黄金、白银、贵金属 |
| 资源周期/化工 | 约3只 | 有色、化工、资源品、涨价链 |
| 大消费 | 约3只 | 消费中具备趋势和基本面支撑的股票 |

明确排除板块：

- 金融
- 房地产
- 建筑
- 外贸
- 农业/养殖
- 公用环保/交通
- 教育/服务

这些排除板块不进入候选池和交易推荐，但未来宏观或市场全局分析模块可以观察其指数表现。

## 3. 总体架构

模块应按独立子系统组织，建议目录如下：

```text
modules/trading-assistant/
├─ server.js
├─ trading-assistant.html
├─ data/
│  ├─ trading-assistant.json
│  ├─ trading-assistant.js
│  ├─ trading-assistant-recommendation-tracking.json
│  ├─ trading-assistant-strategy-log.json
│  └─ trading-assistant-universe-cache.json
├─ scripts/
│  ├─ refresh-trading-assistant.js
│  ├─ audit-trading-assistant-data.js
│  ├─ update-strategy-review.js
│  └─ scheduled-refresh-trading-assistant.ps1
└─ logs/
```

系统运行模式分为两种：

| 模式 | 用途 | 能力 |
|---|---|---|
| 本地模式 | 开发、调试、人工即时刷新 | Node服务提供API，页面可点击刷新真实数据，可写入剔除确认 |
| GitHub Pages模式 | 外网自测访问 | 静态页面读取已发布数据，数据由GitHub Actions定时刷新并部署 |

## 4. 页面设计

Web页面应至少包含以下页面：

| 页面 | 解决的问题 | 核心内容 |
|---|---|---|
| 01 今日行动清单 | 今天最需要看什么、怎么处理 | 交易准备池、重点跟踪池、暂不交易数量；分板块行动建议 |
| 02 候选池 | 哪些股票值得看、为什么 | 候选股来源、板块命中、入池理由 |
| 03 评分排序 | 哪些股票优先级更高 | 分数、趋势、资金、财报、公告、风险、策略投票 |
| 04 买卖点触发 | 是否接近可执行买点 | 建仓区间、买点类型、止损、止盈、触发条件、级别确认 |
| 05 交易计划 | 如果要操作，怎么做 | 首仓、加仓、减仓、止损、止盈、下一步动作 |
| 06 数据审计与复盘 | 数据是否可信、策略是否需要修正 | 数据覆盖率、失败源、不可用字段、策略复盘 |
| 07 剔除确认 | 哪些股票要移出候选池 | 拟剔除理由、用户确认后才停止追踪 |
| 08 推荐追踪 | 历史推荐是否经得起验证 | 推荐日起至当前的价格走势、收益、回撤、计划检验 |
| 09 全量数据 | 便于二次判断和排错 | 每只候选股完整结构化数据 |

页面原则：

- 每只股票必须展示理由、依据、风险和下一步建议。
- 不只给结论，必须展示支持结论的数据和逻辑。
- 分板块展示，科技板块放在最前面。
- 数据缺失必须显示，不允许伪造。
- GitHub Pages静态模式下，刷新按钮只重新读取最新已发布数据；真正刷新由GitHub Actions执行。

## 5. 数据来源

系统应优先使用公开、可验证的数据源，并设计多源降级。

### 5.1 行情与股票池

优先级：

1. 东方财富全市场行情、板块和成分股。
2. 新浪财经板块节点和个股行情。
3. 本地股票池缓存。

要求：

- 如果东方财富失败，自动切换新浪财经。
- 如果新浪板块不提供官方板块涨跌幅和成交额，可以用成分股真实行情聚合计算。
- 如果所有全市场源失败，不应覆盖旧快照，应提示刷新失败。

### 5.2 K线与技术数据

优先级：

1. 东方财富日K、30分钟K、5分钟K。
2. 新浪K线备用源。
3. 腾讯K线第二备用源。

要求：

- 日K用于趋势阶段、均线、波动、支撑压力。
- 30分钟K和5分钟K用于低级别买点确认。
- 如果某个级别K线缺失，应降低买点确认强度，并在数据审计里展示。

### 5.3 财报数据

来源：

- 东方财富财务指标、三表或公开财报摘要。

用途：

- 判断营收、净利润、毛利率、现金流、负债等是否支持股价逻辑。
- 防止只因概念或热点入池。

要求：

- 财报数据用于加分或扣分。
- 财报缺失时不得假设基本面良好。

### 5.4 公告与事件

来源：

- 东方财富公告。
- 巨潮资讯公告备用源。

用途：

- 识别重大合同、业绩预告、回购、减持、定增、问询函、处罚、风险提示等。

要求：

- 正面公告可以加分，但不能替代趋势和风险收益比。
- 负面公告需要显性扣分或进入暂不交易。

### 5.5 资金流

来源：

- 东方财富资金流字段。
- 东方财富实时行情字段。
- 新浪市场资金流或其它可用公开源。

要求：

- 资金流是重要参考项，但不是硬性条件。
- 若资金流无法获取，必须在页面写明“资金流缺失”，不加分也不扣分。
- 不允许因为资金流缺失而伪造主力净流入。

## 6. 选股逻辑

股票进入候选池，不只因为命中板块。推荐逻辑应包括：

```text
板块方向匹配
+ 趋势结构成立
+ 缠论买点接近或确认
+ 量价关系健康
+ 相对强弱突出
+ 基本面有支撑
+ 事件催化清晰
+ 资金行为不冲突
+ 风险收益比合适
+ 能形成可执行交易计划
= 进入推荐或重点跟踪
```

### 6.1 板块方向

判断股票所属板块是否命中用户关注方向，并根据板块权重分配候选名额。

科技板块权重最高，其它保留板块保留约3只。

### 6.2 趋势结构

需要判断：

- 中短期均线是否多头排列。
- 股价是否站上关键均线。
- 是否处于上升趋势、平台突破、趋势修复、回踩确认。
- 是否短期涨幅过大、存在追高风险。

### 6.3 缠论与买卖点

需要吸收缠论作为技术面判断框架，关注：

- 底分型、顶分型。
- 笔、线段、中枢。
- 离开中枢后的回抽确认。
- 一买、二买、三买或类二买、类三买。
- 背驰或类背驰。
- 多级别联立，日线看方向，30分钟和5分钟看触发。

输出不能只写“可以买”，必须写：

- 买点类型。
- 当前是否确认。
- 建仓区间。
- 触发条件。
- 失效条件。
- 止损点。
- 止盈观察位。

### 6.4 量价关系

重点判断：

- 突破是否放量。
- 回调是否缩量。
- 放量上涨后是否滞涨。
- 大跌是否伴随放量。
- 成交额是否足够支持交易执行。

### 6.5 相对强弱

需要比较：

- 股票相对大盘。
- 股票相对行业。
- 股票相对同候选池其它股票。

优先选择板块强、个股更强、回调时抗跌的股票。

### 6.6 基本面支撑

至少检查：

- 营收增长。
- 净利润增长。
- 毛利率变化。
- 现金流质量。
- 负债压力。
- 是否有真实订单、产能、客户、产品或业绩兑现。

只有概念、没有业绩或能力支撑的股票应降低权重。

### 6.7 事件催化

加分事件包括：

- 财报超预期。
- 政策支持。
- 行业涨价。
- 产品突破。
- 订单落地。
- 产业链断供造成替代机会。
- 回购、股权激励、并购重组等。

负面事件包括：

- 减持。
- 监管问询。
- 业绩暴雷。
- 重大诉讼。
- 负面新闻。
- 解禁压力。

### 6.8 风险收益比

推荐必须给出：

- 当前价。
- 建仓区间。
- 止损价。
- 止盈观察价。
- 预期收益空间。
- 预计亏损空间。
- 风险收益比。

如果上方压力过近、止损过远，不应进入交易准备池。

## 7. 状态分层

候选股应分为以下状态：

| 状态 | 含义 | 操作建议 |
|---|---|---|
| 交易准备池 | 评分、趋势、买点、风险收益比基本满足 | 等触发条件确认后可考虑首仓 |
| 重点跟踪池 | 方向较好，但买点或低级别确认不足 | 等回踩、突破或低级别确认 |
| 观察池 | 有一定逻辑，但条件不足 | 保留观察，不执行交易 |
| 暂不交易 | 存在技术、基本面、公告或风险硬约束 | 不交易，等待重新入池或剔除 |

每日行动清单优先展示交易准备池和重点跟踪池。

## 8. 交易计划输出

每只候选股至少输出：

- 股票名称和代码。
- 所属关注板块。
- 当前价格和涨跌幅。
- 入池理由。
- 加分依据。
- 扣分或风险。
- 买点类型。
- 建仓区间。
- 首仓比例。
- 加仓条件。
- 止损价。
- 止盈观察位。
- 下一触发条件。
- 下一步建议。
- 跟踪周期。
- 数据来源。

示例结构：

```json
{
  "code": "000000",
  "name": "示例股票",
  "focus": {
    "area": "科技",
```

## 9. 剔除确认机制

候选池需要连续性，不能每天刷新后直接删除股票。

规则：

1. 如果某只股票上一轮在候选池，本轮不再入池，进入“拟剔除清单”。
2. 页面展示拟剔除理由、上一轮状态、上一轮关键数据。
3. 用户确认后，股票才进入已剔除记录。
4. 如果用户选择继续跟踪，后续刷新应保留并重新分析。
5. 被确认剔除的股票在推荐追踪页置灰并排到最后，不删除历史记录。

GitHub Pages静态模式不能写入剔除确认，剔除确认应在本地模式完成后再推送。

## 10. 推荐追踪机制

推荐追踪用于验证系统历史推荐的有效性。

规则：

1. 当前候选池中的股票都应进入推荐追踪池。
2. 每只股票记录首次推荐日、首次推荐价、首次理由、首次交易计划。
3. 每次刷新后追加价格、状态、评分、交易计划。
4. 页面默认折叠，每只股票一行显示名称、推荐日期、推荐股价、现价、累计收益、状态。
5. 展开后显示从推荐日起到当前查看日的价格曲线、累计收益、最大涨幅、最大回撤、是否触发止损或止盈观察。
6. 如果股票被用户确认剔除，停止更新，但保留历史记录，置灰并排到最后。
7. 推荐追踪池中 active 股票应与当前候选池一致。

## 11. 策略复盘与自我优化

系统需要有自我复盘机制，但不能每天因为单只股票表现不符合预期就机械修改策略。

原则：

- 复盘是一个阶段性机制，而不是每日即时修复。
- 以月度、季度、半年等周期观察整体表现。
- 复盘目标是让系统大方向更正确，而不是追求每笔交易精确。
- 偏差原因可能来自宏观、行业、个股、消息、技术误判或不可控因素。
- 系统只生成优化建议，必须让用户二次确认后才执行策略调整。

复盘输出：

- 样本数量。
- 各状态分层平均表现。
- 命中和失效较多的规则。
- 需要调整的候选规则。
- 建议调整原因。
- 建议调整方式。
- 状态：待用户确认。

## 12. 数据审计要求

每次刷新后必须生成数据审计结果。

审计内容：

- 刷新时间。
- 板块数量。
- 股票池数量。
- 候选股数量。
- 今日行动清单数量。
- 各状态数量。
- 日K覆盖。
- 30分钟K覆盖。
- 5分钟K覆盖。
- 财报覆盖。
- 公告覆盖。
- 资金流覆盖。
- 失败数据源。
- 无法填充字段。
- 候选池与推荐追踪池是否一致。
- 是否命中排除板块。

如果失败源存在，页面必须具体显示，例如：

```text
失败源 2 个：
1. 东方财富概念板块：fetch failed
2. 东方财富行业板块：fetch failed
```

## 13. 刷新机制

### 13.1 本地刷新

本地运行Node服务：

```powershell
node modules/trading-assistant/server.js
```

页面点击“刷新真实数据”后：

1. 调用 `/api/refresh`。
2. 后台运行刷新脚本。
3. 拉取行情、K线、财报、公告、资金流。
4. 重新计算候选池和交易计划。
5. 更新推荐追踪和连续性剔除。
6. 写入 `data/trading-assistant.json` 和 `data/trading-assistant.js`。
7. 页面显示刷新成功或失败明细。

### 13.2 GitHub Pages刷新

GitHub Pages只托管静态页面，不能直接运行后台刷新脚本。

解决方式：

- 使用GitHub Actions定时运行刷新脚本。
- 刷新后构建静态页面。
- 自动部署到GitHub Pages。

当前定时刷新：

| UTC cron | 中国时间 | 说明 |
|---|---|---|
| `10 1 * * 1-5` | 周一至周五 09:10 | 盘前/开盘后初筛 |
| `45 4 * * 1-5` | 周一至周五 12:45 | 午间复核 |
| `40 7 * * 1-5` | 周一至周五 15:40 | 盘后复盘 |

GitHub Pages页面上的刷新按钮只重新读取已发布的最新数据，不触发真实数据刷新。

## 14. API设计

本地Node服务至少提供：

| API | 方法 | 说明 |
|---|---|---|
| `/api/status` | GET | 查询刷新状态、最近刷新时间、最近错误和输出 |
| `/api/snapshot` | GET | 返回当前交易助理快照 |
| `/api/refresh` | POST | 启动后台刷新 |
| `/api/removal/confirm` | POST | 确认剔除股票 |
| `/api/removal/keep` | POST | 继续跟踪股票 |

API要求：

- 支持CORS。
- 刷新过程异步执行。
- 防止重复刷新并发冲突。
- 刷新超时后返回错误。
- 错误信息应可展示给用户。

### 21.2 对这些项目的评价

这些项目的共同优点：

1. 都强调数据、策略、执行或回测的分层。
2. 都不是只看单一指标，而是把数据处理、信号生成、风险控制和结果评估放在一条链路里。
3. 高 star 和长期维护说明它们的工程思路被较多用户验证。
4. 成熟项目都强调风险提示，不把策略输出包装成确定性收益。

这些项目与本系统的关键差异：

1. 它们多偏“量化研究框架”或“交易机器人”，本系统偏“可解释的 AI 股票经纪助手”。
2. 它们通常需要用户自己写策略、准备数据或接交易所/券商，本系统需要直接输出人能读懂的候选池、理由和下一步操作。
3. 它们更重回测、模型和执行，本系统当前更重选股逻辑、入池解释、买卖点触发、人工确认和持续追踪。
4. 它们的输出通常是信号或订单，本系统输出必须包含判断依据、数据来源、风险边界和二次判断材料。

## 22. 与参考项目的选股逻辑差异

### 22.1 Qlib 与本系统

Qlib 的典型思路：

```text
准备因子数据
训练机器学习模型
预测未来收益或排序分数
组合优化
回测验证
执行或模拟执行
```

本系统的思路：

```text
板块过滤
多源行情和基本面校验
趋势与缠论买点判断
资金和事件辅助验证
风险收益比过滤
输出可解释交易计划
持续追踪推荐后表现
```

差异：

- Qlib 更偏模型驱动，适合因子研究和批量预测。
- 本系统更偏规则和解释驱动，适合用户每天阅读、判断和执行。
- Qlib 的入池逻辑通常来自模型分数或因子排序；本系统的入池逻辑必须同时满足板块、趋势、基本面、风险收益比和可操作性。

### 22.2 vn.py 与本系统

vn.py 的典型思路：

```text
连接交易接口
接收行情事件
运行策略
生成订单
风控检查
实盘或模拟执行
```

本系统的思路：

```text
扫描股票
生成候选池
给出交易辅助建议
等待用户人工确认
记录推荐表现
```

差异：

- vn.py 是交易系统开发框架，本系统是交易前决策辅助。
- vn.py 关注订单、账户、网关、实盘环境；本系统关注推荐理由、买点和风险。
- 本系统明确不接券商、不自动下单。

### 22.3 Backtrader 与本系统

Backtrader 的典型思路：

```text
定义策略规则
导入历史数据
模拟买卖
计算收益、回撤、夏普等指标
优化参数
```

本系统的思路：

```text
实时或准实时刷新数据
形成当日候选池
输出当下可执行或待观察计划
后续推荐追踪
```

差异：

- Backtrader 强在历史验证，本系统强在当日决策展示。
- 本系统目前不做完整历史回测，但通过推荐追踪逐步积累真实表现样本。
- 后续可以把推荐追踪数据导入回测框架，验证入池规则和买卖点规则。

### 22.4 Freqtrade 与本系统

Freqtrade 的典型思路：

```text
交易所数据
策略信号
自动或模拟交易
仓位管理
WebUI/Telegram控制
回测和机器学习优化
```

本系统的思路：

```text
A股公开数据
人工可读推荐
买卖点和交易计划
用户确认
不自动交易
```

差异：

- Freqtrade 是机器人，适合加密货币自动交易。
- 本系统是助理，适合 A股人工决策辅助。
- Freqtrade 的策略输出偏自动执行，本系统必须把“为什么推荐、为什么不推荐、下一步看什么”写清楚。

### 22.5 CZSC 与本系统

CZSC 的典型思路：

```text
K线标准化
识别分型
生成笔和线段
识别中枢
判断背驰
形成缠论信号
```

本系统的思路：

```text
使用缠论作为技术面框架之一
结合趋势、量价、基本面、公告、资金和板块
给出买点观察或交易准备建议
```

差异：

- CZSC 更专注缠论技术分析。
- 本系统不把缠论作为唯一依据，而是把它放进交易计划的买卖点触发层。
- 即使出现类二买或类三买，如果基本面、公告、风险收益比不合格，也不能进入交易准备池。

### 22.6 AKShare 与本系统

AKShare 的典型思路：

```text
提供多类财经公开数据接口
用户自行调用、清洗、分析
```

本系统的思路：

```text
自动调用公开数据
自动降级和审计
自动生成候选池和交易计划
```

差异：

- AKShare 是数据接口库，不负责选股和交易建议。
- 本系统是应用层工具，数据只是输入，重点是分析、推荐、解释和追踪。
- 未来如果需要更广泛的数据，AKShare 可作为 Python 数据层候选。

## 23. 我们的入池逻辑与参考项目的具体差异

### 23.1 参考项目常见入池逻辑

不同项目常见做法如下：

| 类型 | 常见入池方式 | 优点 | 缺点 |
|---|---|---|---|
| AI量化模型 | 按预测收益、IC、因子分数排序 | 覆盖面广，可批量筛选 | 解释性弱，对普通用户不直观 |
| 回测框架 | 按策略信号触发买入 | 规则明确，可历史验证 | 需要用户自己定义策略，缺少基本面和事件解释 |
| 自动交易机器人 | 按技术指标和风控规则触发 | 执行闭环强 | 容易变成机械交易，不适合当前人工确认目标 |
| 缠论工具 | 按分型、笔、中枢、背驰形成信号 | 买卖点结构清晰 | 如果单独使用，容易忽略基本面和外部风险 |
| 数据接口库 | 不提供入池逻辑 | 数据覆盖广 | 需要另行构建分析层 |

### 23.2 本系统的入池逻辑

本系统采用“先过滤，再评分，再分层，再追踪”的逻辑。

第一层：板块和股票池过滤

```text
必须命中保留板块
不能命中明确排除板块
成交额和流动性不能过低
基础行情数据必须可获取
```

第二层：基础质量过滤

```text
财报不能出现明显硬伤
公告不能存在重大负面硬约束
价格结构不能处于明显破位下跌
风险收益比不能明显不合格
```

第三层：综合评分

评分参考：

- 板块权重。
- 个股趋势。
- 量价结构。
- 缠论买点。
- 相对强弱。
- 财报质量。
- 公告和事件。
- 资金流。
- 风险收益比。
- 数据完整性。

第四层：状态分层

```text
交易准备池：可以形成明确交易计划，等待或已经接近触发。
重点跟踪池：方向较好，但买点或低级别确认还不完整。
观察池：有逻辑，但条件不足。
暂不交易：存在硬约束或风险收益比不合格。
```

第五层：推荐追踪

```text
入池股票进入推荐追踪。
后续每天记录价格、状态、评分和计划变化。
踢出候选池必须用户确认。
```

### 23.3 为什么不直接采用“模型排名前N”入池

原因：

1. 用户要的是可解释交易助理，不是黑箱排序。
2. A股题材、政策、公告、资金和板块联动强，仅靠模型分数容易误导。
3. 用户需要知道“为什么看、什么时候买、错了怎么办”，模型排名无法完整回答。
4. 当前阶段真实可获得数据存在不稳定性，必须把数据缺失和失败源纳入判断。

### 23.4 为什么不直接采用“技术指标触发买入”入池

原因：

1. 单一技术指标容易在震荡市产生大量假信号。
2. 技术信号不等于公司有真实业绩或产业逻辑。
3. 用户强调不能只有概念，也不能只有技术图形。
4. 本系统必须把技术面、基本面、事件、资金、风险收益比一起呈现。

### 23.5 为什么不直接采用“缠论信号”入池

原因：

1. 缠论适合买卖点判断，但不应单独决定股票池。
2. 一只股票即使出现类二买，如果处于排除板块、财报恶化或公告负面，也不能推荐。
3. 缠论结构有一定主观性，需要结合多级别和其它数据交叉验证。
4. 本系统将缠论定位为“买卖点触发系统”，而不是唯一选股系统。

## 24. 我们的推荐逻辑相对开源项目的增强点

本系统相对参考项目，面向用户使用结果做了以下增强：

1. 输出不只是信号，而是完整操作建议。
2. 每只股票都显示推荐理由、加分项、扣分项和下一步观察。
3. 候选池与推荐追踪池保持一致，方便验证历史推荐。
4. 剔除候选池需要用户确认，避免系统每天刷新导致推荐断裂。
5. 缺失数据不参与硬性评分，并在页面中显性说明。
6. 资金流是重要参考，但缺失时不伪造、不默认加分或降级。
7. GitHub Pages 可以免费外网访问，Actions 每天自动刷新三次。
8. 本地开发和线上自测分离，便于持续迭代。

## 25. 推荐给其它AI工具的复刻重点

如果其它 AI 工具要按本文档复刻类似系统，必须特别注意：

1. 不要只复刻 UI，要复刻“入池逻辑 + 交易计划 + 推荐追踪 + 数据审计”。
2. 不要只用技术指标，要同时考虑板块、趋势、缠论、财报、公告、资金、风险收益比。
3. 不要把机器学习模型分数直接当推荐结论。
4. 不要因为数据源失败而填充虚构值。
5. 推荐必须分层：交易准备、重点跟踪、观察、暂不交易。
6. 每只股票必须能回答：
   - 为什么入池？
   - 当前能不能买？
   - 买点是什么？
   - 建仓区间在哪里？
   - 止损在哪里？
   - 止盈看哪里？
   - 什么情况下加仓？
   - 什么情况下剔除？
7. 推荐必须被持续追踪，用真实走势检验系统判断。
